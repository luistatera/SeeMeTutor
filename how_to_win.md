## The winning formula for Live Agents (checklist)

### Minimum bar (if you miss any, don’t bother)

* **Live voice loop**: low-latency talk/listen, interruptions, short confirmations.
* **Live vision loop**: camera feed → agent references what it sees *now* (not “upload an image”).
* **Grounding**: citations from a small curated knowledge base (docs), or tool outputs; plus “I don’t know / can’t see that” handling. ([Gemini Live Agent Challenge][1])
* **Cloud proof + diagram**: explicit in judging. ([Gemini Live Agent Challenge][1])
* **Demo video shows the product working** (not slides). ([Gemini Live Agent Challenge][1])

### Differentiators (what separates finalists)

* **Agent actually *acts***: executes steps via tools (calendar, browser actions, device controls, forms), like Jayu’s “interact with interfaces” vibe. ([developers.googleblog.com][2])
* **Narrow but intense workflow**: 1 job, 10/10 execution (vs 5 jobs, 3/10).
* **Safety rails**: confirmations before irreversible actions, fail-soft behavior, and explicit uncertainty.

### Demo script that wins (3 minutes)

1. 10s: the problem + why live multimodal is required (not text).
2. 90s: one flawless “hero flow” (voice + vision + tool actions).
3. 30s: failure handling (noise, wrong object, missing info).
4. 30s: architecture diagram + cloud proof + repo/IaC mention. ([Gemini Live Agent Challenge][1])

## 3 highest-probability project directions (optimized for Live Agents)

### 1) **“Live Repair Copilot” (field technician / operator)**

**What it does:** user points camera at equipment/PCB/label; agent speaks step-by-step troubleshooting, asks clarifying questions, logs findings, and pulls grounded steps from manuals/known issues.
**Why it fits:** Live vision+voice is essential; strong “real world” impact like Vite Vere; easy to demo; aligns with “avoid hallucinations” via grounding. ([developers.googleblog.com][2])
**Must deliver to win:**

* Real-time camera + voice loop
* Grounded retrieval (small doc set) + cite snippets on-screen
* “Create a repair note” action (saves to DB) + export summary

**Weaknesses:** hard to demo without physical gear; hallucination risk if visuals are ambiguous; needs tight scope.

---

### 2) **“Live Accessibility Coach” (independent living assistant)**

**What it does:** “What am I looking at?” + “What’s the next step?” for simple tasks (making tea, sorting meds, reading labels), with safety checks.
**Why it fits:** Mirrors proven winner pattern (Vite Vere) but pushes *live* interaction deeper (interruptions, continuous guidance). ([developers.googleblog.com][2])
**Must deliver to win:**

* Continuous guidance loop (“ok now move camera closer…”)
* Safety gating (“before you take medication, confirm label matches…”)
* Clear persona/voice + calm UX

**Weaknesses:** judges may see it as “too similar” to past winners if not differentiated; must avoid medical advice claims.

---

### 3) **“Live Meeting Wingman” (real-time conversation assistant)**

**What it does:** listens to live audio (meeting/call), tracks decisions + action items, and speaks up only when asked (“what did we agree?”). Optional: vision on shared screen to reference a slide/table.
**Why it fits:** Live audio-first; strong demo; high utility; can show robust architecture + grounding (“here’s what was said at 01:32”).
**Must deliver to win:**

* Real-time transcription + state tracking
* “Ask the agent verbally” mid-meeting
* Output: action list + email draft + calendar holds

**Weaknesses:** privacy concerns; speech diarization is hard; can feel less “vision-based” unless you intentionally include screen vision.
