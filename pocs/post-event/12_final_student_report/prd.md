# POC 12 — Final Student Report: Mini PRD

## Why This Matters

A live tutoring session is ephemeral. To deliver lasting educational value, the student (or parent) needs a tangible artifact. While POC 08 introduced a basic A2A (Agent-to-Agent) Reflection text summary, this new POC aggregates the rich, multimodal outputs from *all* previous POCs into a single, beautiful "End of Class Report."

This proves that SeeMe Tutor isn't just a voice bot, but a comprehensive learning platform that retains context, visualizes progress, and provides actionable takeaways.

---

## What Information Should the Report Deliver? (The "Nuggets")

Based on the analysis of POCs 01-11, the report should filter out the noise (e.g., latency metrics, guardrail triggers) and focus purely on educational value. The Final Report should consist of the following sections:

### 1. The Executive Summary (Sourced from POC 02 & 08)

* **Session Objective:** What was the explicitly agreed-upon pedagogical objective at the start of the session? (From POC 02: Proactive Vision). This is the anchor metric for the entire report.
* **Objective Status:** Did the student achieve the stated session objective?
* **Mastered Concepts:** What secondary concepts did the student successfully learn or demonstrate? (Sourced via the Reflection Agent).

### 2. The Artifact Board (Sourced from POC 04 & 05)

* **Whiteboard Snapshot:** A consolidated view of all the Markdown notes generated by the tutor during the session (from POC 04). This acts as the student's "study guide."
* **Fact Check / Citations:** A list of verified facts and links that the tutor looked up during the session (from POC 05: Search Grounding). Example: "Things we looked up today."

### 3. Dynamic Learning Modules (Topic-Specific)

* **Language & Fluency (If applicable):** L2 immersion rate and vocabulary/grammar corrections (from POC 03). Only appears if the session involved language learning.
* **Formula & Concept Review (If applicable):** Extracts the core formulas or technical definitions the student struggled with.
* *The report structure should automatically adapt to the subject matter (e.g., University-level Calculus vs. High School German).*

### 4. Focus & Engagement (Generic)

* **Active vs. Idle Time:** Session duration minus time spent in "Away" or "Idle" modes (applicable to any subject).
* **Interactive Moments (Barge-ins):** A metric showing how often the student actively interrupted or asked questions.

### 5. Next Steps

* **Action Items:** 1-2 concrete things to practice before the next session, generated by the Reflection Agent (from POC 08).

## Architecture Flow

1. **Incremental Persistence:** Every time a nugget is generated (Goal set, Whiteboard note sent, Citation displayed, Language switch, or Idle check-in), the Orchestrator instantly writes/appends that event to the session document in Firestore.
2. **Session Abandonment Detection:** Since students often close the browser without saying goodbye, the backend tracks the last activity timestamp.
    * If the WebSocket closes (normally or via drop) AND no reconnect happens within X minutes (e.g., 5 mins).
    * *Or* if a server-side cron job sweeps for "stale" sessions (no activity for 15+ mins).
3. **A2A Generation (Async):** Once a session is deemed "completed/abandoned", a background worker (or Cloud Task) triggers.
    * It gathers the raw transcript from Firestore.
    * It passes the transcript to the Reflection Agent (heavy text model like Gemini 1.5 Pro).
4. **Final JSON Assembly:** The background worker merges the A2A qualitative JSON (Mastered, Struggles, Next Steps) with the deterministically saved arrays (notes, citations, metrics) into the final `StudentReport` object in Firestore. The JSON schema must remain generic and subject-agnostic.
5. **Storage & Display:** The UI Dashboard listens to Firestore. When the report finishes generating, it becomes visible to the student.

---

## What "Done" Looks Like (Must-Haves)

* [ ] **M1:** State is persisted incrementally: closing the browser mid-session does not lose whiteboard notes or citations gathered so far.
* [ ] **M2:** The backend reliably detects session end/abandonment (e.g. no reconnect within 5 minutes) and triggers the Reflection Agent asynchronously.
* [ ] **M3:** The report accurately displays at least 1 Search Citation triggered during the session.
* [ ] **M4:** The report summarizes the explicitly agreed-upon **Session Objective** from the start of the call and deterministically states whether it was met.
* [ ] **M5:** The JSON payload structure is **Generic and Subject-Agnostic** (it adapts to language, math, or history dynamically) and handles null arrays cleanly.

## Integration Readiness Metrics (When is it ready for Main App?)

This POC cannot be merged into the main application until it consistently hits these performance and stability benchmarks:

| Metric | Target | Why it matters |
| :--- | :--- | :--- |
| **JSON Parse Success** | 100% | The generic JSON output from the Reflection Agent must successfully parse and map to our TypeScript/Pydantic schemas every time, without hallucinating malformed arrays. |
| **A2A Latency (End to DB Write)** | < 12s (P90) | The student shouldn't be staring at a "Generating report..." spinner for a minute. The transition from closing the class to viewing the dashboard must feel snappy. |
| **Data Loss Prevention** | 0% Loss | Simulating a browser crash mid-session must result in *all* previously completed events (whiteboard notes, citations) appearing in the final report via the background worker. |
| **Language Agnosticism** | Pass | The A2A Reflection Agent must generate a valid summary for a Math session (with formulas) and a German session (with vocabulary) using the *same* backend code and prompt. |
